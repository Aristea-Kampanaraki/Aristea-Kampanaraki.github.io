WOWI Sales Analysis

SQL Server Case Study — Business Intelligence Analyst Simulation (CFI BIDA Certification)

🗄️ SQL Server 💻 Visual Studio Code 🔌 mssql Extension ⭐ Star Schema 📊 Data Analysis 🎓 BIDA Certification
❮ ❯

A case-based SQL course from Corporate Finance Institute's Business Intelligence & Data Analyst (BIDA) certification, simulating a BI Analyst role at Worldwide Importers Ltd (WOWI) — a fictional novelty goods importer and wholesale distributor. Working directly against a pre-built star schema data warehouse in SQL Server, the brief was to answer real business questions from finance, marketing, delivery, and product teams — not textbook exercises, but the kind of ambiguous, constraint-laden asks a BI Analyst actually gets handed.

Case study & database: Corporate Finance Institute (CFI), BIDA Certification. Queries, analysis, and this write-up are original work.

The Challenge

The role wasn't "write SQL" — it was act as a Business Intelligence Analyst and translate what finance, marketing, and operations teams actually asked for into queries against WOWI's SQL Server database. None of the requests arrived as a query; they arrived as business language with constraints buried inside it. The marketing team's Halloween ask — "ship a low-cost item for free" — came with a catch stated almost as an aside: most of the cheapest items are just packaging. Translating that meant building an explicit exclusion list (box, bag, carton) into the filter, not just sorting by price. The operations team's delivery question — "group deliveries by postcode and buying group instead of by individual store" — wasn't a lookup either; it only became answerable once reframed as a threshold on an aggregate, not a row-level filter.

Finance's questions pushed the translation further: "why is profit lower this year" isn't a metric to compute, it's a comparison to investigate — which meant working out that the real answer was fewer months of recorded data, not lower per-month performance. Every task followed the same shape: read what a team actually needed to know, then decide which SQL pattern made that answer come out correctly and unambiguously.

Key Techniques

  • Sentinel-Aware Counting: Known-customer counts explicitly exclude the 'Unknown' placeholder row (WHERE Customer <> 'unknown') rather than assuming COUNT(*) reflects real customers — the same sentinel-vs-NULL discipline that matters in any warehouse with placeholder dimension rows.

  • Multi-Condition Campaign Targeting: The Halloween promo brief layered three separate constraints — product type (LIKE '%mug%' OR LIKE '%shirt%'), colour, and a packaging-material exclusion list — into one filter, then used TOP 1 ... ORDER BY with an explicit tie-break column to pick a single winning product deterministically.

  • GROUP BY + HAVING for Threshold Detection: Finding postcodes with more than 3 Wingtip Toys stores needed a threshold on an aggregate, not a row-level filter — the kind of query where reaching for WHERE instead of HAVING would silently return the wrong thing.

  • EXCEPT for Gap Detection: Finding fiscal months with zero recorded sales used a set-subtraction (EXCEPT) between "all possible months" and "months that appear in factSale," instead of a NOT IN subquery or a LEFT JOIN ... WHERE NULL — a cleaner statement of the actual question being asked.

  • RIGHT JOIN Audit Pattern: Auditing chiller products for zero sales-to-date required preserving every row in dimStockItem even when no matching row exists in factSale — a standard JOIN would silently drop exactly the zero-sales products the audit was trying to find.

  • Two Reusable "Current Year" Approaches, Compared Deliberately: WHERE [Fiscal Year] = YEAR(GETDATE()) is simple but returns nothing if the latest data hasn't loaded yet; anchoring instead to MAX([Fiscal Year]) FROM factSale always resolves to the latest year with actual data — the difference between a query that looks correct in testing and one that's safe to schedule against a production warehouse.

  • Multi-Table Attribution: Identifying the top product/salesperson combination for a fiscal year needed a 4-table join (factSale, dimDate, dimStockItem, dimEmployee) collapsed to a single ranked row — attribution questions like this are where a star schema's payoff shows up directly.

  • Ranking an Aggregate: "Which sales territory has the highest population" needed dimCity rows summed up to the territory level first — GROUP BY [Sales Territory] with SUM([Latest Recorded Population]) — before ORDER BY ... DESC could rank anything, since no single row in the table holds a territory's total population on its own.

  • Business Formula → Explicit Precision: Markup ((retail price − unit price) / unit price) is a plain arithmetic expression, but wrapped in CAST(... AS decimal(10,4)) deliberately — left as the default division result, SQL Server's implicit typing can silently round a markup ratio to far less precision than a real percentage needs.

The Outcome

Eight stakeholder-style questions across four teams, answered with SQL that goes beyond single-table filtering: set operators for gap detection, threshold aggregation, join-direction chosen deliberately for audit completeness, and reusable queries that were written to keep working outside the training environment they were built in. Completed as part of Corporate Finance Institute's BIDA certification.
