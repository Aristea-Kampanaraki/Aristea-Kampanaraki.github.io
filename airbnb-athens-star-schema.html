Airbnb Athens Data Warehouse

SQL Server Star Schema — Analysis of the Athens, Greece Airbnb Market

🗄️ SQL Server 🐍 Python · pyodbc ⭐ Star Schema 🔄 SCD Type 1 🏘️ PropTech 🌍 Inside Airbnb
❮ ❯

A SQL Server data warehouse built from Inside Airbnb's Athens, Greece listings — designed to answer market-analysis questions (which neighbourhoods perform best, what drives ROI, where regulatory risk concentrates) rather than to be a generic Airbnb clone dataset. Raw quarterly CSVs are staged into SQL Server, merged into two permanent history tables, and transformed into a star schema (6 dimensions, 2 fact tables) — snapshot-based, quarterly-updated, and ready to connect to Power BI.

Data: Inside Airbnb · CC BY 4.0 · insideairbnb.com

The Challenge

SQL Server Express doesn't support BULK INSERT ... WITH (FORMAT='CSV'), which rules out the simplest ingestion path for a dataset with 90 raw columns spread across three separate quarterly file types. The challenge was building a pipeline that stays reproducible every quarter without hand-editing a single query — and doing it on the free tier of SQL Server, not a paid edition with the missing feature built in.

The answer was a hybrid path, not a single tool: listings loads through a Python/pyodbc script that replicates the exact column-type overrides a manual import wizard would apply, making quarterly reloads fully scriptable. reviews and neighbourhoods — both far larger, less schema-sensitive files — load through SSMS's Import Flat File wizard instead, since native bulk copy is roughly 10–30x faster than a row-by-row Python insert at that row count (~800K rows for reviews alone). Two ingestion paths, each earning its place on its own performance and reproducibility merits — not one workaround stretched to cover everything.

On top of that ingestion layer sits a star schema that has to survive being rebuilt every quarter without corrupting anything: dimensions built with ROW_NUMBER()-based SCD Type 1 deduplication (a pattern found through an actual primary-key-violation build failure, not designed in the abstract), sentinel keys instead of NULL foreign keys, and every schema change guarded with an idempotent IF NOT EXISTS check so the same script is always safe to re-run.

Key Decisions

  • Snapshot-Based History: listings/reviews never truncated, tagged snapshot_date — SCD Type 1 history stays analyzable even after a rebuild.

  • SCD Type 1 via ROW_NUMBER(): Found through an actual PK-violation build failure — host 37177 lost Superhost status between snapshots, producing two rows for the same host_id. Not a theoretical design choice.

  • Time-Varying Attributes → Facts: host_is_superhost lives in FactListings, not DimHost — enables Lost/Gained/Retained trend analysis.

  • Sentinel Keys, Never NULL FKs: id = 0 = Unknown. Power BI relationships handle a sentinel far better than a NULL.

  • Two Missing-Data Patterns: Review scores → NULL (AVG() excludes them); response rates → -1 sentinel (must filter explicitly).

  • DimLocation From Reference File: Sourced from neighbourhoods.csv, sidestepping a real Greek-text encoding bug in listings entirely.

  • property_category (added to DimPropertyType) — clusters the 48 granular property types into 5 investor-facing groups, implemented with an idempotent IF NOT EXISTS guard.

  • DimDate Lives in SQL Server: Not locked into Power BI/DAX — available to any tool that connects to the database.

  • Fully Recoverable: Every table is derived, never hand-edited. Delete + rebuild regenerates everything cleanly.

Read the full reasoning behind each decision — worked examples, attribute placement tables, and trade-offs — in the Design Decisions doc on GitHub.

The Outcome

This project shows how a quarterly data release from Inside Airbnb can become a ready-to-analyze market model — without a paid database tier, without manually fixing the schema every time new data lands, and without losing the ability to answer the questions that actually matter later: which neighbourhoods perform best, how reliable is that number, and what's changed since last quarter.

The schema is ready for loading into Power BI, where the tables get connected into a working model.
